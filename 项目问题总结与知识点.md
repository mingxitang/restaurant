# 餐厅点餐系统项目问题总结与知识点

本文档总结本项目开发过程中遇到的主要非样式类问题，以及对应的原因、修复方式和可提炼的知识点。

## 1. CORS 配置启动失败

### 问题现象

后端启动时报错：

```text
Could not resolve placeholder 'app.cors.allowed-origins'
Error creating bean with name 'webConfig'
```

### 原因分析

`WebConfig` 中使用：

```java
@Value("${app.cors.allowed-origins}")
private List<String> allowedOrigins;
```

直接注入 YAML 中的列表配置时，Spring 没有正确解析该占位符，导致 Bean 创建失败。

### 解决方式

改为通过 `Environment` 读取配置，并提供默认值：

```java
String origins = environment.getProperty(
    "app.cors.allowed-origins",
    "http://localhost:5173,http://127.0.0.1:5173"
);
```

### 知识点

- Spring Boot 中 `@Value` 适合读取简单字符串配置。
- YAML 列表配置建议使用 `@ConfigurationProperties` 或 `Environment`。
- 对非核心配置提供默认值，可以提高系统启动稳定性。

## 2. MySQL 表名 `user` 导致 SQL 风险

### 问题现象

登录或查询用户时可能出现 SQL 执行异常。

### 原因分析

项目中用户表命名为：

```sql
user
```

`user` 在 MySQL 中容易与系统语义或关键字产生冲突。Mapper 中直接写：

```sql
FROM user u
```

存在解析风险。

### 解决方式

统一改为反引号包裹：

```sql
FROM `user` u
```

建表、外键、插入语句也统一修改：

```sql
CREATE TABLE `user` (...)
REFERENCES `user`(user_id)
INSERT INTO `user`(...)
```

### 知识点

- 数据库表名应尽量避免使用关键字或系统保留词。
- 如果必须使用，应在 SQL 中用反引号包裹。
- MyBatis XML 中的 SQL 与数据库语法强绑定，应保持命名规范。

## 3. 默认账号密码哈希不匹配

### 问题现象

使用默认账号登录失败。

### 原因分析

初始化 SQL 中写入的 BCrypt 哈希值不能匹配明文密码 `123456`。

Spring Security 登录时使用：

```java
passwordEncoder.matches(rawPassword, encodedPassword)
```

如果哈希不匹配，认证会失败。

### 解决方式

使用 `BCryptPasswordEncoder` 重新生成 `123456` 对应的哈希，并更新 `data.sql`。

### 知识点

- Spring Security 默认推荐使用 BCrypt 存储密码。
- 数据库中不能存明文密码。
- 初始化账号的密码哈希必须与登录时使用的 `PasswordEncoder` 算法一致。

## 4. 数据库缺少 `remark` 字段导致加菜失败

### 问题现象

提交加菜时报错：

```text
Unknown column 'remark' in 'field list'
```

### 原因分析

后端代码已经向 `order_detail.remark` 字段写入备注，但旧数据库中的 `order_detail` 表没有该字段。

### 解决方式

新增迁移 SQL：

```sql
ALTER TABLE order_detail
ADD COLUMN remark VARCHAR(255) NULL AFTER unit_price;
```

并同步更新 `schema.sql`，保证新建数据库时也包含该字段。

### 知识点

- 修改实体类和 Mapper 后，必须同步修改数据库表结构。
- 对已有数据库，应提供迁移脚本，而不是只修改建表脚本。
- `target/classes` 是编译输出目录，源文件应维护在 `src/main/resources/sql` 下。

## 5. 退最后一份菜失败

### 问题现象

退菜时，如果退掉某菜品的最后一份，会出现失败。

### 原因分析

表结构中有约束：

```sql
quantity INT NOT NULL CHECK (quantity > 0)
```

原逻辑先执行：

```sql
quantity = GREATEST(quantity - #{quantity}, 0)
```

当退菜数量等于已点数量时，`quantity` 会被更新为 `0`，违反 `CHECK (quantity > 0)`。

### 解决方式

改为两种情况处理：

```sql
-- 如果退完，直接删除明细
DELETE FROM order_detail
WHERE order_id = #{orderId}
  AND dish_id = #{dishId}
  AND quantity <= #{quantity}

-- 如果只是部分退菜，再扣减数量
UPDATE order_detail
SET quantity = quantity - #{quantity}
WHERE order_id = #{orderId}
  AND dish_id = #{dishId}
  AND quantity > #{quantity}
```

### 知识点

- 有数据库约束时，业务更新逻辑必须遵守约束。
- 数量型明细表中，`0` 数量通常不应保留，应删除该明细。
- 退菜、退款、库存回滚应放在事务中处理。

## 6. 同一桌台加菜生成多个订单

### 问题现象

同一个桌台每次加菜都会生成新订单，导致：

- 已点菜品列表出现多个相同菜品。
- 一个桌台对应多张未结订单。
- 结账和退菜逻辑变复杂。

### 原因分析

原下单逻辑每次都执行：

```java
orderMapper.insert(order);
```

没有判断该桌台是否已经存在未结账订单。

### 解决方式

新增查询当前桌台未结订单：

```sql
SELECT *
FROM orders
WHERE table_id = #{tableId}
  AND status = 'PENDING'
LIMIT 1
```

如果存在，则复用该订单；如果不存在，才创建新订单。

### 知识点

- 餐饮业务中常见规则是“一桌一单”。
- 加菜不应创建新订单，而应追加到当前桌台账单。
- 业务约束应尽量放在后端保证，不能只依赖前端表现。

## 7. 同一订单中相同菜品重复插入失败

### 问题现象

同一订单中再次添加同一道菜时，加菜失败或明细异常。

### 原因分析

`order_detail` 使用联合主键：

```sql
PRIMARY KEY (order_id, dish_id)
```

同一订单中，同一道菜只能有一条明细。直接再次插入相同 `(order_id, dish_id)` 会主键冲突。

### 解决方式

后端改为：

1. 先尝试更新已有明细数量。
2. 如果更新行数为 0，再插入新明细。

```java
int updated = orderMapper.addDetailQuantity(detail);
if (updated == 0) {
    orderMapper.insertDetail(detail);
}
```

### 知识点

- 联合主键可防止重复明细。
- 对“同菜品加数量”的业务，应使用更新数量，而不是重复插入。
- MyBatis 可以通过返回影响行数判断是否需要执行后续插入。

## 8. 使用 `ON DUPLICATE KEY UPDATE` 后兼容性和可控性不足

### 问题现象

曾尝试使用：

```sql
INSERT ... ON DUPLICATE KEY UPDATE
```

处理同菜品加菜，但后续排查和控制不够直观。

### 原因分析

该写法依赖 MySQL 特性，虽然简洁，但对于课程设计和业务调试来说，可读性不如显式判断。

### 解决方式

改成 Service 层显式控制：

```java
int updated = orderMapper.addDetailQuantity(detail);
if (updated == 0) {
    orderMapper.insertDetail(detail);
}
```

### 知识点

- 数据库特有语法可以提高效率，但会降低跨数据库兼容性。
- 课程设计中，清晰的业务流程有时比一条复杂 SQL 更合适。
- Service 层负责业务编排，Mapper 层负责单一 SQL 操作。

## 9. 退菜后前端状态不刷新

### 问题现象

退菜后，已点菜品列表不会立即变化，有时需要再点一次菜品才刷新。

### 原因分析

退菜接口只插入退菜记录和回滚库存，没有同步修改订单明细；前端刷新后仍然查到原来的订单明细。

### 解决方式

退菜时同时处理：

- 插入退菜记录。
- 扣减或删除订单明细。
- 扣减订单金额。
- 根据 `stock_action` 决定是否回库。

### 知识点

- 退菜不是单纯记录日志，还要影响订单明细和金额。
- 业务流水表与当前状态表要同时维护。
- 退菜操作应使用事务，保证多表更新一致。

## 10. 点菜、退菜后页面“刷屏”

### 问题现象

点菜或退菜后，整个首页像重新刷新一样，视觉体验差。

### 原因分析

前端每次操作后都调用完整的：

```js
load()
```

导致首页统计、桌台、订单、菜品、分类、月度统计全部重新加载。

### 解决方式

改为局部刷新：

```js
refreshAfterTableMutation()
```

只刷新必要数据：

- 当前桌台订单。
- 菜品库存。
- 桌台状态。
- 首页统计。

并且刷新当前桌台订单时，不先清空列表，而是新数据回来后直接替换。

### 知识点

- 前端不应在局部操作后重载整个页面数据。
- 局部刷新能减少闪烁和布局跳动。
- 异步更新时避免先清空列表，可以改善用户体验。

## 11. 加菜失败时前端没有错误提示

### 问题现象

提交加菜失败时，页面没有明显反馈，看起来像按钮失效。

### 原因分析

前端提交订单时没有 `try/catch`：

```js
await createOrder(...)
```

接口失败后错误没有显示到页面。

### 解决方式

增加异常捕获：

```js
try {
    await createOrder(...)
} catch (error) {
    message.value = error.message || '提交加菜失败'
}
```

### 知识点

- 前端调用接口必须处理异常。
- 后端错误信息应通过统一响应格式返回。
- 用户操作失败时，应给出明确提示，便于定位问题。

## 12. 未结账关台与订单状态的关系

### 问题现象

用户希望即使未结账也能关台。

### 原因分析

桌位状态和订单支付状态是两个不同概念：

- 桌位状态表示桌台是否可用。
- 订单状态表示账单是否结算。

原交互把二者绑定得太紧。

### 解决方式

新增“关台”操作：

- 只释放桌位为 `FREE`。
- 不修改订单状态。
- 未结订单仍保留在今日订单中。

结账操作则会：

- 支付订单。
- 更新订单为 `COMPLETED`。
- 释放桌位。

### 知识点

- 桌位状态和订单状态应解耦。
- “关台”不等于“结账”。
- 餐饮业务中可能存在桌台释放但账单未结的情况。

## 13. 今日订单未结统计

### 问题现象

未结账关台后，需要在今日订单处看到未结数量。

### 原因分析

原首页只展示今日订单总数，没有区分已结和未结。

### 解决方式

前端基于今日订单计算：

```js
const todayUnpaidCount = computed(() =>
  todayOrders.value.filter((order) => !isPaid(order)).length
)
```

并在今日订单卡片显示：

```text
未结 N
```

### 知识点

- 统计口径要清晰：订单总数、已收金额、未收金额是不同指标。
- 简单统计可由前端基于已加载数据计算。
- 复杂统计或大数据量场景应交给后端 SQL 聚合。

## 14. 今日营业额统计口径调整

### 问题现象

用户希望今日营业额区分：

- 今日已收多少。
- 今日未收多少。
- 今日总共多少。

### 原因分析

原本只显示后端返回的 `todayRevenue`，它只统计已支付/已完成订单金额，不包含未结账订单。

### 解决方式

前端基于今日订单计算：

```js
todayPaidAmount
todayUnpaidAmount
todayTotalAmount
```

### 知识点

- 财务类数据必须明确统计口径。
- 已收金额应来自已支付订单。
- 未收金额应来自未支付订单。
- 合计金额 = 已收 + 未收。

## 15. 多选退菜与菜品合并展示

### 问题现象

同一桌台同一道菜可能出现多条记录，用户希望按菜名合并，并支持多选退菜。

### 原因分析

前端最初按订单明细逐条展示，未按菜品聚合。

### 解决方式

前端将当前桌台所有订单明细按 `dishId` 分组合并：

- 合并数量。
- 合并金额。
- 合并备注。
- 保留来源订单明细，用于退菜时拆分处理。

多选退菜时，按来源订单逐个调用退菜接口。

### 知识点

- 展示层可以做聚合展示，但业务操作仍需要保留原始来源数据。
- 合并展示不能丢失订单 ID、菜品 ID、单价等业务字段。
- 批量操作应逐项校验数量是否合法。

## 16. Promise.all 快速失败导致整页数据丢失

### 问题现象

首页加载时，如果 `dashboard()`、`listTables()`、`listOrders()`、`listDishes()`、`listCategories()`、`monthlyRevenue()` 中任意一个接口失败，整页数据全部空白，所有请求结果都被丢弃。

### 原因分析

原代码使用 `Promise.all` 并行请求：

```js
const [dashboardData, tableData, orderData, dishData, categoryData, monthData] =
  await Promise.all([dashboard(), listTables(), listOrders(), listDishes(), listCategories(), monthlyRevenue()])
```

`Promise.all` 的语义是"全成功或全失败"：只要一个 Promise reject，整体立即 reject，其他已成功的结果也不会返回。

### 解决方式

改为 `Promise.allSettled`：

```js
const results = await Promise.allSettled([
  dashboard(), listTables(), listOrders(), listDishes(), listCategories(), monthlyRevenue()
])
data.value = results[0].status === 'fulfilled' ? results[0].value : {}
tables.value = results[1].status === 'fulfilled' ? results[1].value : []
// ...
```

每个 Promise 独立处理结果，失败的提供默认值，不影响其他成功的请求。

同样修复了以下位置：
- 首页 `refreshAfterTableMutation()`
- 菜品统计页 `ReportsView.vue`

### 知识点

- `Promise.all` 适用场景：所有请求都必须成功的强依赖场景（如多步骤事务）。
- `Promise.allSettled` 适用场景：独立数据源的并行加载，某个失败不影响其他。
- 前端加载多个独立接口时，优先考虑 `allSettled` 配合默认值兜底。
- 单个接口超时或 500 不应导致整个页面不可用。

## 17. 厨师角色权限前后端不匹配

### 问题现象

- 厨师登录后，侧边导航显示"身份管理"、"菜品管理"等所有菜单项。
- 厨师可以手动输入 URL 访问 `/users` 等无权限页面。
- 但后端 `@PreAuthorize("hasRole('管理员')")` 会拦截请求，页面显示空白或 403。
- 厨师在首页能够看到"开台"按钮，但操作后会失败。

### 原因分析

前端路由和导航没有做角色过滤：

1. 侧边导航 `nav[]` 数组对所有角色展示全部菜单。
2. 路由器 `beforeEach` 没有检查 `meta.roles`。
3. 路由配置缺少 `meta.roles` 字段。
4. 前端页面中的操作按钮没有按角色隐藏（如开台、加菜、退菜）。
5. 后端菜品/分类/报表接口仅允许管理员，厨师无法管理。

### 解决方式

前端四层控制：

1. **路由定义**：每个路由增加 `meta: { roles: ['管理员', ...] }`。
2. **路由守卫**：`beforeEach` 中检查用户角色，无权限重定向 `/dashboard`。
3. **导航过滤**：`Layout.vue` 中的 `nav[]` 用 `.filter()` 按角色过滤。
4. **页面按钮控制**：`DashboardView.vue` 中通过 `isChef` / `isAdmin` computed 属性条件渲染。

后端权限放开：菜品 CRUD、分类 CRUD、报表接口增加 `'厨师'` 角色，桌位查询增加 `'厨师','顾客'`。

### 知识点

- RBAC 必须前后端双重控制：前端负责 UI 显示/隐藏，后端负责接口鉴权。
- 路由守卫防止直接输入 URL 绕过导航栏。
- 前端按钮条件渲染防止误操作，但不能替代后端鉴权。
- `v-if="!isChef"` 可以简洁地按角色控制 Vue 模板中的 UI 区域。
- 后端 `@PreAuthorize` 注解应与前端路由角色保持一致。

## 18. 新订单首次加菜金额累加失败

### 问题现象

桌台首次点餐时（新建订单），提交加菜后订单金额异常或为 null。

### 原因分析

原逻辑使用 `orderMapper.increaseTotal(orderId, total)` 增加订单金额：

```sql
UPDATE orders SET total_amount = total_amount + #{amount} WHERE order_id = #{orderId}
```

新建订单的 `total_amount` 初始值为 `NULL`，`NULL + X = NULL`，导致金额始终为 NULL。

### 解决方式

区分两种情况处理：

```java
if (order.getTotalAmount() != null && order.getTotalAmount().compareTo(BigDecimal.ZERO) > 0) {
    orderMapper.increaseTotal(order.getOrderId(), total);   // 已有金额：累加
} else {
    orderMapper.updateTotal(order.getOrderId(), total);     // 新订单：直接设置
}
```

同时移除了无效的 `boolean newOrder` 变量，改为直接判断 `order.getTotalAmount()` 是否已有值。

### 知识点

- SQL 中 `NULL + X` 结果为 `NULL`，不是 `X`。
- 新建记录的初始值场景和已有值的累加场景需要不同的 SQL 操作。
- `BigDecimal` 比较用 `compareTo`，不能用 `==` 或 `>`。
- 移除无效的中间变量 `newOrder`，直接根据数据状态判断，代码更清晰。

## 19. Spring 6 null-safety 编译警告

### 问题现象

IDE 对以下代码发出黄色警告：

```java
// JwtAuthenticationFilter.java
protected void doFilterInternal(HttpServletRequest request, ...)

// WebConfig.java
public void addCorsMappings(CorsRegistry registry)
```

警告内容：方法参数可能为 null，但未标注可空性。

### 原因分析

从 Spring 6 / Spring Boot 3 开始，框架引入了更严格的 null-safety 注解检查。以下方法签名中参数被框架标记为 `@NonNull` 或可空性未明确：

- `OncePerRequestFilter.doFilterInternal()` 的参数在框架中标注了 `@Nullable`，实现类未标注会导致警告。
- `WebMvcConfigurer.addCorsMappings()` 同理。

### 解决方式

添加 `@NonNull` 注解和 `@SuppressWarnings("null")`：

```java
import org.springframework.lang.NonNull;

protected void doFilterInternal(
    @NonNull HttpServletRequest request,
    @NonNull HttpServletResponse response,
    @NonNull FilterChain filterChain) ...

@SuppressWarnings("null")
public void addCorsMappings(@NonNull CorsRegistry registry)
```

### 知识点

- Spring 6 使用 `org.springframework.lang.NonNull` / `@Nullable` 进行 null-safety 静态分析。
- 覆盖父类方法时，参数可空性标注应保持兼容。
- `@SuppressWarnings("null")` 可以抑制无法消除的 null-safety 警告。
- 这些注解主要影响 IDE 检查和编译期静态分析，不影响运行时行为。

## 20. 反结账时占用桌台导致同台未结账点单被吞

### 问题现象

对某张桌台的已支付订单执行"反结账"后：
- 该桌台状态被强制刷回 `OCCUPIED`（占用）。
- 如果该桌台当时已有新的未结账点单，这些点单的显示和状态可能被覆盖或"吞掉"。
- 用户无法正常查看和管理原桌台当前正在进行的订单。

### 原因分析

后端 `OrderService.unpay()` 方法在回退支付状态后，无条件执行：

```java
tableInfoMapper.updateStatus(order.getTableId(), "OCCUPIED");
```

问题在于反结账的业务语义被理解错了：
- 反结账 = 将某笔**历史订单**从已支付回退为待支付。
- 桌台状态 = 该桌台**当前**是否有人正在用餐（由当前有效的 PENDING 订单决定）。

这两个概念是解耦的。反结账操作不应强行改变桌台状态。如果桌台当时空闲（FREE）且只有这一笔被反结的订单，桌台自然应该变为 OCCUPIED；但如果桌台已有其他新点单，桌台本来就是 OCCUPIED，强行再设一次可能干扰前端渲染，甚至覆盖正确状态。

### 解决方式

移除 `unpay()` 方法中的 `tableInfoMapper.updateStatus(...)` 调用：

```java
@Transactional
public void unpay(Long orderId) {
    Order order = orderMapper.findById(orderId);
    // ... 校验 ...
    orderMapper.unpay(orderId);
    // 不修改桌台状态 — 桌台状态由实际存在的有效订单决定
}
```

### 知识点

- 反结账只回退支付状态，不是"重新开台"。
- 桌台状态应由桌台上**所有有效订单的集合**决定，而不是由单个订单操作决定。
- 状态管理应避免"副作用链"：反结账 → 改桌台状态 → 影响其他订单显示。
- 业务操作应遵循"最小影响原则"，只修改需要修改的数据。

## 21. 桌台管理入口分散导致卡片膨胀

### 问题现象

- 每张桌台卡片上都有一个"改名"按钮，增加了卡片高度和视觉复杂度。
- 添加新桌台需要点击 `floor-head` 区域的 `+ 添加桌台` 按钮，展开一个独立表单。
- 添加、改名、删除三个桌台管理操作分散在两处入口（卡片内 + 头部区域），操作路径不统一。

### 原因分析

桌台管理操作（增、改、删）是低频操作，不应与点餐（开台、加菜、退菜、结账）等高频操作混在一起放置。直接在每张卡片上放置操作按钮会：
- 增大卡片尺寸，减少桌面空间利用。
- 用户操作点餐时这些按钮是视觉噪音。
- 添加桌台的入口与改名入口分离，不一致。

### 解决方式

将增、改、删桌台操作统一为一个"编辑模式"：

1. 首页标题 `h2` 右侧放置"编辑"按钮，点击切换编辑模式。
2. 编辑模式下：
   - 每张桌台卡片变为可编辑表单（名称、区域、人数），显示"保存"和"删除"按钮。
   - 末尾新增虚线边框的"添加"卡片，用于新增桌台。
3. 非编辑模式下，卡片只显示桌台信息，无任何管理按钮。

前端核心实现：

```js
const showEditMode = ref(false)
const editForm = reactive({})

function toggleEditMode() {
  showEditMode.value = !showEditMode.value
  if (showEditMode.value) {
    tables.value.forEach(table => {
      editForm[table.tableId] = {
        tableName: table.tableName || '',
        area: table.area || '',
        capacity: table.capacity || 4
      }
    })
  } else {
    Object.keys(editForm).forEach(key => delete editForm[key])
  }
}
```

### 知识点

- 低频管理操作与高频业务操作应有明确的界面分界。
- "编辑模式"是一个通用 UI 模式：正常状态只读展示，切换后进入可编辑状态，适用于批量管理和配置场景。
- Vue 3 的 `reactive({})` 对象可以按 ID 动态添加/删除属性，天然适合"多个同类表单"的编辑场景。
- 退出编辑模式时清理表单数据，避免下次进入时显示脏数据。

## 总结

本项目中的问题主要集中在以下几类：

1. 配置问题：如 CORS 配置读取失败。
2. 数据库结构问题：如缺少字段、表名冲突、约束冲突。
3. 业务规则问题：如一桌一单、关台与结账解耦、新订单首次金额设置、反结账与桌台状态解耦。
4. 多表一致性问题：如退菜同时影响库存、订单明细、订单金额和退菜记录。
5. 前后端交互问题：如接口失败无提示、局部刷新不合理、`Promise.all` 快速失败。
6. 统计口径问题：如已收、未收、合计金额的区分。
7. 权限控制问题：如厨师角色前后端权限不匹配、路由守卫缺失。
8. 代码规范问题：如 SQL 金额 NULL 累加陷阱、Spring 6 null-safety 编译警告。
9. UI 架构问题：如桌台管理入口分散、低频操作与高频操作混杂、编辑模式设计。

通过这些问题可以提炼出一个核心经验：数据库课程设计不能只停留在建表和查询，还必须结合真实业务流程，考虑事务一致性、状态流转、统计口径和前后端协作。同时，前端容错性（如 `Promise.allSettled`）、RBAC 双重控制、SQL 空值陷阱和框架版本升级带来的规范要求也是实际开发中不可忽视的环节。
